# Render Deploy Guide

This project is prepared for Render Blueprint deployment with `render.yaml`.

## Prerequisites

- A GitHub/GitLab/Bitbucket repository containing this project.
- A Render account connected to that Git provider.
- A Google OAuth client for web sign-in.

## Local Preflight

Run these before pushing:

```powershell
npm.cmd run import:data
npm.cmd test
npm.cmd run typecheck
npm.cmd run build
npm.cmd run check:render
npm.cmd audit --omit=dev
```

For local multiplayer smoke while the local server is running:

```powershell
$env:ALLOW_DEV_AUTH='true'
npm.cmd start
npm.cmd run smoke:multiplayer
```

## Push To Git

Render deploys from a Git repository. If this folder is not pushed yet:

```powershell
git add .
git commit -m "Prepare RTTS MMO demo for Render"
git remote add origin <YOUR_GIT_REPOSITORY_URL>
git push -u origin main
```

Use the actual branch name if it is not `main`.

## Create The Render Blueprint

1. Open the Render Dashboard.
2. Choose **New +**.
3. Choose **Blueprint**.
4. Select the repository that contains this project.
5. Confirm that Render detects the root `render.yaml`.
6. Fill the required sync:false values.

The Blueprint creates:

- Web Service: `rtts-mmo-demo`
- Render Postgres: `rtts-mmo-demo-db`

The service uses:

- Runtime: Node
- Node version: `22.12.0`
- Build command: `npm ci && npm run import:data && npm run build`
- Start command: `npm start`
- Health check: `/api/health`

## Required Environment Variables

The Blueprint already defines these keys:

- `NODE_ENV=production`
- `NODE_VERSION=22.12.0`
- `SESSION_SECRET` generated by Render
- `DATABASE_URL` from `rtts-mmo-demo-db`

Fill these during Blueprint creation:

- `APP_ORIGIN=https://<your-render-service>.onrender.com`
- `GOOGLE_CLIENT_ID=<Google OAuth client id>`
- `GOOGLE_CLIENT_SECRET=<Google OAuth client secret>`

Do not set `ALLOW_DEV_AUTH=true` in production.

## Google OAuth Redirect URI

In the Google Cloud Console OAuth client, add this authorized redirect URI:

```text
https://<your-render-service>.onrender.com/auth/google/callback
```

If you use a custom domain, add that callback URI too:

```text
https://<your-domain>/auth/google/callback
```

## Post-Deploy Verification

After Render finishes the first deploy:

1. Open:

```text
https://<your-render-service>.onrender.com/api/health
```

Expected:

```json
{
  "ok": true,
  "shardId": "demo-1"
}
```

2. Open the site root and sign in with Google.
3. Confirm the tactical UI shows:
   - `Shard demo-1`
   - connection `open`
   - Round/Turn/Clock/AP/HP HUD
   - visible actors on the map
4. Open a second browser profile or private window, sign in with another Google account, and confirm both clients join the same shard.

## Common Failure Points

- `Google OAuth is not configured.`
  - `GOOGLE_CLIENT_ID` or `GOOGLE_CLIENT_SECRET` is missing.
- `Invalid OAuth state.`
  - Cookies are blocked or `APP_ORIGIN` does not match the deployed URL.
- Google says `redirect_uri_mismatch`.
  - Add the exact Render callback URL to the Google OAuth client.
- Health check fails.
  - Confirm the service binds to Render's `PORT`. This app reads `process.env.PORT`.
- WebSocket does not connect.
  - Confirm the browser is using the deployed HTTPS origin so it opens `wss://<host>/ws`.
